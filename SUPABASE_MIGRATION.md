# ShinobiX — Supabase Migration Guide

Migrates persistent game storage from **Upstash Redis** to **Supabase Postgres**
and runs the API under **cPanel / Phusion Passenger** instead of Vercel.

> **Status: COMPLETE.** This migration has already been run in production. The
> one-off migration/import scripts (`migrate-upstash-*.mjs`, `import-*.mjs`) and
> the `ioredis` dependency have been removed from the repo. This guide is kept for
> historical reference and the schema/setup notes below; see git history for the
> original migration script source.

---

## What Changed

| Before | After |
|--------|-------|
| `@vercel/kv` (Upstash Redis) | `@supabase/supabase-js` via `api/_storage.ts` |
| Vercel serverless functions | Express on cPanel/Passenger |
| `api/` files import from `@vercel/kv` | `api/` files import from `./_storage.js` |

All handler files are **unchanged** except for the single import line.

## Supabase Setup

### 1. Run the schema

In the Supabase SQL Editor, run `supabase-schema.sql`.  
This creates:

| Object | Purpose |
|--------|---------|
| `public.kv_store` | Main key-value store |
| `kv_store_expires_at_idx` | Fast expired-row cleanup |
| `kv_store_key_pattern_idx` | Fast `LIKE` pattern scans |
| `kv_set_nx()` | Atomic set-if-not-exists (PvP locks) |
| `kv_hset()` | Atomic hash-field merge |
| `kv_hdel()` | Atomic hash-field removal |
| `kv_delete_expired()` | Cleanup helper |

### 2. Verify in Supabase

```sql
select count(*) from public.kv_store;
select public.kv_delete_expired();
```

### 3. Set server environment variables

In `/home/fatedreu/shinobix-api/.env`:

```env
SUPABASE_URL=https://soaychxshtbgwujhytsf.supabase.co
SUPABASE_SERVICE_ROLE_KEY=eyJ...
ADMIN_PASSWORD=your-admin-password
OPENAI_API_KEY=sk-...        # only needed for /generate-image
```

**Never** expose `SUPABASE_SERVICE_ROLE_KEY` in frontend code,
`VITE_*` variables, or public directories.

## Data Migration (Upstash → Supabase)

### Prerequisites

You need both sets of credentials:

- Upstash: `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN`
  (from Upstash dashboard → your database → REST API)
- Supabase: `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY`
  (from Supabase dashboard → Project Settings → API)

The migration script runs **locally** — it connects to both services
over HTTPS, no server access required.

### Run the migration

**PowerShell:**
```powershell
cd "C:\Users\Tyler R\source\repos\NinjaK"

$env:UPSTASH_REDIS_REST_URL   = "https://your-db.upstash.io"
$env:UPSTASH_REDIS_REST_TOKEN = "AXxx..."
$env:SUPABASE_URL             = "https://soaychxshtbgwujhytsf.supabase.co"
$env:SUPABASE_SERVICE_ROLE_KEY= "eyJ..."
# optional: $env:KEY_PATTERN = "save:*"   # migrate only player saves

node scripts\migrate-upstash-to-supabase.mjs
```

**bash/Mac:**
```bash
UPSTASH_REDIS_REST_URL="https://your-db.upstash.io" \
UPSTASH_REDIS_REST_TOKEN="AXxx..." \
SUPABASE_URL="https://soaychxshtbgwujhytsf.supabase.co" \
SUPABASE_SERVICE_ROLE_KEY="eyJ..." \
node scripts/migrate-upstash-to-supabase.mjs
```

The script:
1. SCANs all Upstash keys (batches of 200)
2. Pipelines TYPE + TTL for each key
3. Pipelines GET (string keys) or HGETALL (hash keys)
4. Upserts into `kv_store` in batches of 100
5. Writes `upstash-backup-YYYY-MM-DD.json` (local backup)

Supported Redis types: **string**, **hash**.  
Unsupported types (list, set, zset): logged as warnings and skipped.

### Verify migration

```sql
-- Total rows migrated
select count(*) from public.kv_store;

-- Player saves
select key, updated_at from public.kv_store where key like 'save:%' order by updated_at desc limit 10;

-- Image maps
select key from public.kv_store where key like 'shared:%';

-- Admin content
select key from public.kv_store where key like 'save:admin%';
```

## Ongoing Maintenance

### Expired key cleanup

PvP sessions, heartbeats, and challenge records have TTLs (10 min, 60 s, 2 min).
They are lazily deleted on read by the adapter. For proactive cleanup:

```sql
-- Run manually in Supabase SQL Editor
select public.kv_delete_expired();
```

To automate, enable **pg_cron** in Supabase (Database → Extensions) then:

```sql
select cron.schedule(
    'kv-cleanup',
    '*/5 * * * *',
    $$ select public.kv_delete_expired(); $$
);
```

### Storage monitoring

```sql
-- Row counts by key prefix
select
    split_part(key, ':', 1) as prefix,
    count(*) as rows,
    count(*) filter (where expires_at is not null and expires_at > now()) as with_ttl
from public.kv_store
group by prefix
order by rows desc;
```

## Rollback

If you need to revert to Upstash:

1. Restore the original import in each API file:
   ```typescript
   import { kv } from '@vercel/kv';
   ```
2. Set `KV_REST_API_URL` and `KV_REST_API_TOKEN` environment variables on Vercel.
3. Redeploy.

The backup file (`upstash-backup-YYYY-MM-DD.json`) contains all original data.
