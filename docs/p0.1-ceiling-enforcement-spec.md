# P0.1 — Server-Authoritative PvP Power Ceiling

**Status:** READ-ONLY SPEC (no code edits). Implementation-ready; do one sub-part at a time.
**Design pillar:** balanced PvP — the shared power CEILING is held by server-enforced rules, never bought/grinded/RNG'd. "Ceiling, not speed."
**Goal of P0.1:** move the ceiling guards that are today only client-side onto the server so a forged save (public repo) cannot exceed them.

All line numbers are "near here" — confirm before editing.

## 2. Sub-part 1 — Enforce the bloodline point budget server-side

### 2.1 Problem (cited)
- Budget math lives only on the client: `pointBudgetForRank` (`jutsu-points.ts:18` → B/Academy 7, A 10, S 11), `tagPointValue` (`:26`), `jutsuPoints`/`bloodlinePoints` (`:79,83`). `BloodlineMaker.tsx:213-214,258-259` is the only enforcement.
- Save normalizer clamps per-jutsu numerics (`[name].ts`:771-809) but never recomputes `bloodlinePoints` or rejects over-budget; `out.totalPoints` is cosmetic, clamped `[0,20]` (`:762`).
- PvP `sanitizeJutsuList` (`session.ts:187-248`) clamps each jutsu but not the per-bloodline budget.
- Net: a forged save ships a bloodline whose jutsus individually pass numeric clamps but together exceed the rank budget → extra Stun/Copy/Wound/amp tags → power above the ceiling.

### 2.2 New module `api/_jutsu-points.ts`
Verbatim numeric port of `shinobij.client/src/lib/jutsu-points.ts` (same duplicate-and-parity-test pattern as `_pet-stat-ceil.ts`). Port `pointBudgetForRank` (S 11 / A 10 / else 7), `tagPointValue` (table `jutsu-points.ts:26-48`; reuse `CAPPED_AMP_TAGS` from `_tags.ts:73` and `ampTagCapForRank` from `move.ts:500` for the rank tag cap S40/AB35/else30), `jutsuPoints`/`bloodlinePoints` incl. structural add-ons (40-AP utility +1; range≥5 +0.5; ground method; nuke +1; cooldown≤1 +0.5 — `:63-75`). Header: "duplicate of client; guarded by cross-build parity test."

### 2.3 Deterministic strip algorithm (clamp, NEVER reject)
`enforceBloodlineBudget(jutsus, rank)`:
1. `budget = pointBudgetForRank(rank)`, `total = bloodlinePoints(jutsus, rank)`. If `total ≤ budget` → return unchanged (**honest no-op, byte-identical**).
2. Else reduce deterministically: iterate jutsus in array order, within each iterate `tags` in array order; removal priority = `(tagPointValue ascending, then index ascending)` — remove lowest-point tags first. Recompute after each removal; stop at `total ≤ budget`. Tags removable even to empty (tagless 60-AP = plain hit). Do NOT touch structural sources (AP/range/method/cooldown) — only tags.
3. Re-stamp `totalPoints = bloodlinePoints(result, rank)`.

Order is fully defined → save-strip and session-strip produce identical results (required for parity).

### 2.4 Insertion A — save normalizer
In `normalizeBloodlineArray` (`[name].ts:743-812`), after the per-jutsu clamp loop (after L809, before `return out`): resolve `rank = out.rank` (validated L749-750, restricted by sub-part 3); `out.jutsus = enforceBloodlineBudget(out.jutsus, rank)`; `out.totalPoints = bloodlinePoints(out.jutsus, rank)` (keep `Math.min(20,…)` display backstop). Covers nested `char.savedBloodlines` (L820) and top-level (L1167).

### 2.5 Insertion B — PvP session
Budget is per-bloodline, so enforce in `resolveEquippedLoadout` (`session.ts:447-478`), not `sanitizeJutsuList`. When pulling `save.savedBloodlines` jutsu (L463-465), run `enforceBloodlineBudget(bl.jutsus, bl.rank)` before inserting into `extra`. Built-in catalog jutsu exempt. Defense-in-depth: closes the window where a pre-existing over-budget save loads into a fight.

### 2.6 Save-compat
Clamp, don't reject — no save 400'd. First POST re-writes at-budget; session also enforces. Honest old-rules saves are already ≤ budget → untouched.

### 2.7 Edge cases
Unknown rank → B (budget 7). Empty/non-array jutsus → no-op. Single over-budget jutsu (Copy3+Mirror3+Stun2=8 on B7) → tags stripped to ≤7. Point values are quarter-increments; `≤` boundary is fine.

### 2.8 Flag / rollout
`BLOODLINE_BUDGET_SERVER` (default OFF) — off skips the call (byte-identical). Honest path already a no-op, so flag mainly de-risks the strip. cPanel: rebuild + commit dist.

### 2.9 Tests
New `api/_jutsu-points.test.ts` (unit) + `api/_jutsu-points-parity.test.ts` (static-text parity vs client). Extend `api/pvp/_sanitize.test.ts` / new `api/save/_bloodline-budget.test.ts`: over-budget B-rank → ≤7; at-budget → deep-equal unchanged; forged 5×Copy/Mirror/Stun red-team → pinned surviving tags; **parity assertion**: same forged input through `sanitizeCharacterSave` AND `resolveEquippedLoadout`+`enforceBloodlineBudget` → identical tag sets.

### 2.10 DoD
`api/_jutsu-points.ts` parity-tested; save + session both clamp deterministically, never reject; flag-off byte-identical; red-team fixture proves trim in both paths.

## 3. Sub-part 2 — Carry `bloodlineRank` through `normalizeJutsu`

### 3.1 Problem
`normalizeJutsu` (`shinobij.client/src/lib/jutsu.ts:15-67`) returns a fixed shape WITHOUT `bloodlineRank`. `getAllJutsus` stamps rank (`App.tsx:1615,1619-1621`) then `.map(normalizeJutsu)` (`:1627`) strips it. Combat reads `jutsu.bloodlineRank` in `woundCapForJutsu` (`move.ts:293-297,760`) and `ampTagCapForRank`/`scaledTagPercent` (`:510-514,614`) → with field absent both collapse to basic (Wound 25 / amp 30) instead of S (35/40) or A (30/35). Wrong in the conservative direction, but inconsistent with PvE (`Arena.tsx:2629,2911` read it directly).

### 3.2 One-field fix
In `normalizeJutsu` return (alongside `updatedAt`/`weatherElement` carry-through L63-66): `...(jutsu.bloodlineRank != null ? { bloodlineRank: jutsu.bloodlineRank } : {})`. Type already has `bloodlineRank?: Rank` (`types/combat.ts:60`). Additive.

### 3.3 Flow into PvP session (server stamp — required)
The per-jutsu objects in `savedBloodlines[i].jutsus` do NOT carry rank — rank lives on `bl.rank`. So in `resolveEquippedLoadout` (`session.ts:463-465`), stamp each jutsu `bloodlineRank: bl.rank` (after sub-part 3 clamps `bl.rank`). Authoritative — does not trust client-sent rank. `sanitizeJutsuList` spreads `{...j}` (L194) so it passes through; confirm nothing deletes it.

### 3.4 Save-compat
Additive. A/S honest bloodlines get their CORRECT (slightly higher) caps — a small balance lift in the intended direction → ship behind a flag.

### 3.5 Edge cases
Mixed loadout fine (undefined → basic). Forged client rank irrelevant — server overwrites from `bl.rank`.

### 3.6 Flag
`BLOODLINE_RANK_CAPS.v1` (default OFF) gates the server stamp (3.3); flag-off = caps stay basic = byte-identical. Client `normalizeJutsu` change is harmless; ship 3.2 + 3.3 together under the flag.

### 3.7 Tests
Client: `normalizeJutsu({…bloodlineRank:'S Rank'})` returns the field. Server `api/pvp/_rank-caps.test.ts`: session from S-rank save → resolved jutsu carry rank → assert S caps (35/40); flag-off basic. Extend `api/_combat-formula-parity.test.ts` to pin `WOUND_CAP_BY_RANK` + `ampTagCapForRank` vs client.

## 4. Sub-part 3 — Gate bloodline `rank` on a real entitlement

### 4.1 Problem
`rank` is self-declared; unknown → B (`[name].ts:748-750`) but any known rank accepted as-is. A forged save sets `rank:'S Rank'` for higher caps (sub-part 2), higher multiplier (S 1.20 / A 1.15 / B 1.10, `_multipliers.ts:113`), and larger budget (sub-part 1).

### 4.2 Constraint
S-rank intentionally locked (Mythic Seals have no faucet). **Do NOT add a faucet.** Clamp rank DOWN to proven entitlement only.

### 4.3 "Proven" (pick per how rank is actually granted — confirm first)
1. **Server-written entitlement field** (best): a forge/upgrade endpoint stamps `char.maxBloodlineRank` / per-bloodline `rankEntitlement`; save POST may never raise it (mirror `createdAt` immutability `[name].ts:368-373`).
2. **Currency-spend proof** at write under lock (don't add new spend flows here).
3. **Conservative fallback (ship first):** clamp `rank` to the value already stored on the existing save — rank is monotonic per bloodline id, never increases via plain save POST; new bloodlines default B. Upgrades come from a dedicated server endpoint (built separately).

### 4.4 Algorithm (replace L749-750)
`rawRank = known(out.rank) else 'B'`; `entitled = provenMaxRank(existingBloodlineById(out.id), char)`; `out.rank = minRank(rawRank, entitled)` (B<A<S, clamp down only; never force-upgrade). Thread prior `savedBloodlines` into `normalizeBloodlineArray` (it currently only gets the array; `sanitizeCharacterSave` already has `exChar`).

### 4.5 Order
Clamp rank FIRST, then budget (sub-part 1) against clamped rank; multiplier (`_multipliers.ts:105`) auto-correct once save clamped.

### 4.6 Save-compat (critical)
Existing legit A/S holders must NOT be downgraded → fallback keys off existing stored rank. New accounts (`FIRST_SAVE_BASELINE_CHARACTER` `:226` has `savedBloodlines:[]`) can only create B. **Risk:** a wiped/migrated save lacking the bloodline would clamp a legit S to B → prefer option 1 for genuinely-granted high rank. **Run a one-off `save:*` audit for legit non-B holders before flipping the flag.**

### 4.7 Edge cases
Match on id; admin saves bypass sanitizer (`[name].ts:1479,1531`); no currency read (hard rule satisfied).

### 4.8 Flag
`BLOODLINE_RANK_ENTITLEMENT.v1` (default OFF). Ship after the save audit.

### 4.9 Tests
`api/save/_bloodline-rank.test.ts`: forged new S no-existing → B; B→S upgrade → B; existing A re-asserted → A; existing A forged S → A; legit A→B downgrade allowed; admin untouched. Integration: clamped rank flows into multiplier (forged-S → 1.10) and budget (7 not 11).

## 5. Sub-part 4 — Server jutsu-power ceiling for pets (ranked ladder)

### 5.1 Problem + key finding
`_pet-stat-ceil.ts` ceilings only hp/atk/def/spd. The **deterministic ranked duel** runs in `api/pet-ladder/_core.ts` → `runPetDuel` → `_duel-sim.ts` (`buildAbility` reads `j.power` L147). `snapshotLadderPet` clamps hp/atk/def/spd (L110-113) but `snapshotJutsu` clamps `power` to flat `clampStat(raw.power,1,1000,80)` (`_core.ts:89`) — not per-rarity. Legit caps: `petStatCaps[rarity].jutsuPower` = standard 320 / rare 360 / legendary 405 / mythic 450 (`pet-stats.ts:28-33`; client `capPetStats` `pet-balance.ts:54`). So a tampered pet seals up to 1000 (~2.2× mythic) into ranked.
**Finding:** the **casual** path (`api/pet/battle-result.ts`) does NOT simulate — trusts client "I won" (bounded by rate limit + daily cap). The deterministic clamp surface is `_core.ts`, NOT `battle-result.ts`. Spec the clamp in `_core.ts snapshotJutsu`; re-assert in save normalizer for hygiene.

### 5.2 Cap
Add to `api/_pet-stat-ceil.ts`:
```ts
export const PET_JUTSU_POWER_CAP: Record<string, number> =
  { standard: 320, rare: 360, legendary: 405, mythic: 450 };
export function petJutsuPowerCeil(rarity: unknown): number {
  return PET_JUTSU_POWER_CAP[String(rarity)] ?? PET_JUTSU_POWER_CAP.mythic;
}
```
Exact mirror (honest pet already ≤ cap → no-op); unknown rarity → mythic (loosest).

### 5.3 Insertion
1. **Ladder (primary):** thread rarity into `snapshotJutsu(raw, rarity)`; at L89 replace flat clamp with per-rarity, preserving 0-power utility (`raw.power<=0 ? 0 : min(cap, max(1, round(power)))`); update call at L103.
2. **Save normalizer (hygiene):** in the pet loop (`[name].ts:584-592`), clamp each `p.jutsus[].power` via `petJutsuPowerCeil(p.rarity)`.

### 5.4 Save-compat / edge
Exact client mirror → honest pet unchanged. Unknown rarity → mythic. 0-power utility stays 0. Casual path out of scope.

### 5.5 Flag
`PET_JUTSU_POWER_CEIL.v1` (default OFF) → off keeps flat 1000.

### 5.6 Tests
Extend `api/_pet-stat-ceil.test.ts` + cross-build parity (server `PET_JUTSU_POWER_CAP` == client `petStatCaps[*].jutsuPower`). Extend `api/pet-ladder/_core.test.ts`: forged `{standard, power:9999}` → 320; `{power:0}` → 0; parity ladder-clamp == save-clamp.

## 6. Sub-part 5 — Budget custom-item bonuses

### 6.1 Problem + key finding
`api/admin/item-review.ts` is approve-only — no stat validation (L50-52). `_multipliers.ts sumEquippedBonus` (L85-97) sums item bonuses for the 5 passive %s + shield; `session.ts:551-558` clamps only those + shield, not raw specialty stats. Save normalizer clamps each `creatorItems[].bonuses[k]` to `[0,1000]` per field (`[name].ts:1156-1161`) — per-field, no per-slot budget.
**Key finding (verify):** item **specialty-stat** bonuses are **INERT in authoritative PvP** — `hydrateCharacterFromSave` uses base save stats (`session.ts:570`); `getOffense`/`getDefense` (`move.ts:275-286`) read un-augmented stats. The CLIENT folds bonuses into a derived display stat (`Arena.tsx:255-266`) but that's not persisted and the server ignores client stats. Live server item surfaces: 5 passive %s + shield (clamped), weapon EP/effect/tags (clamped `session.ts:288-315`), armorQuality→DR (clamped 1.5 `_multipliers.ts:148`). So sub-part 5 = defense-in-depth + storage hygiene + closing the admin approve-without-validation gap — NOT a live specialty-stat exploit fix.

### 6.2 Reference magnitudes (built-in legendaries — the baseline the budget must not undercut)
From `_item-catalog.ts`: legendary armor = 8 specialty @ +30 (=240) + one passive %@1 or shield 100; legendary gloves (hand) = 4 off @+75, 4 def @+30 (=420) + one %@1; hand weapons = one offense ≤+168 + weaponEp ≤30 + effect ≤35; aura `chakra-ring` = maxChakra 150 + ninjutsuOffense 30.

### 6.3 Per-slot budget `budgetItemBonuses(item)`
- passive %s (`damage/absorb/reflect/lifeSteal`) → clamp each to **1**; `shield` ≤ 100 (stricter than current `[0,1000]`; a forged "100% lifesteal" → 1%).
- specialty stats: `total = Σ specialty`; if `> maxSpecialtyTotal[slot]` scale all proportionally (deterministic; mirrors `[name].ts:431-438`). Budgets: armor 240; hand 420; covers weapons (≤168); unknown slot → loosest (hand).
- vitals (maxHp/maxChakra/maxStamina) → `maxVitalBonus` ≥ 150.

### 6.4 Insertion
1. **creatorItems normalizer (save):** replace `[0,1000]` clamp (`[name].ts:1156-1161`) with `budgetItemBonuses(item)`.
2. **Authoritative derivation:** apply `budgetItemBonuses` inside `buildItemLookup`/`sumEquippedBonus` (`_multipliers.ts:50,85`) when wrapping a custom `creatorItems` entry (skip built-in `ITEM_CATALOG`).
3. **admin/item-review.ts:** on approve, run `budgetItemBonuses`; reject-with-error or store-clamped; at minimum `recordAudit` the over-budget delta (already calls it, L55). Confirm where pending items are stored.

### 6.5 Save-compat
Built-ins within budget (no-op). Forged item scaled/clamped, never rejected/removed. Existing approved over-budget items clamped at next session/save.

### 6.6 Edge / flag
No-bonus items no-op; aura vitals covered; unknown slot loosest. `ITEM_BONUS_BUDGET.v1` (default OFF) → off keeps current behavior.

### 6.7 Tests
New `api/pvp/_item-budget.test.ts`: built-in legendary armor/gloves/weapon → identical (pin vs `_item-catalog.ts`); forged 12×1000 specialty → proportional scale; `lifeStealPercent:100` → 1; `shield:99999` → 100. Parity guard: budget caps ≥ richest built-in per slot. admin-review test: over-budget approve rejected/clamped+audited.
